Architecture review · SOAR-like rule/object engine

Make rules and objects meet through a real execution protocol

The current design is close to something useful, but the seams are showing because rule semantics, object semantics, and agent/world ownership semantics are currently mixed together. The clean fix is to introduce a first-class resolver + binding + typed operation layer, then make both objects and rules speak that protocol.

Main recommendation: typed action schemas Kill fallback mutation Make binding explicit Separate objects from rule syntax

1. Executive proposal

Recommended architecture: keep the object flavor, but make the rule engine operate on a data/fact model.

Objects should describe their identity, tags, components, inventory, and state. Rules should not directly mutate objects. Instead, rules should compile into action schemas whose effects are explicit operations such as SetState, Transfer, RevealFromContainer, Consume, and CreateEntity.

1

Entity/object layer

Stores state, tags, components, inventory relations, and names. It does not know rule syntax like tag: or NOT_.

2

Resolver/binding layer

Turns selectors such as “a Drink_Vessel” into concrete entity bindings before action execution.

3

Operation layer

Applies typed effects. Unknown effects fail loudly at compile time or action validation time, not silently during gameplay.

2. What is currently fighting you

ProblemEffects are not a real language yet

Current effects are mostly triples like ("Me", "has", item) or (subject, "has", value). But those triples mean different things depending on subject: possession transfer, state mutation, drinking, retrieval from containers, travel, or generic object mutation.

Because one tuple shape is doing too many jobs, the engine needs coercion and fallback behavior.

ProblemRules resolve objects too late

A tag reference such as tag:Drink_Vessel is resolved at condition/effect time by scanning agent/world collections. That works for the coffee demo, but it becomes fragile when an action needs the same vessel across multiple preconditions and effects.

You want “bind a mug, then keep using that mug,” not “each tuple independently finds the first matching thing.”

ProblemThe object layer depends on the rule layer

Container lookup currently understands tag:-style references. That means object infrastructure knows about rule DSL syntax. This is backwards: containers should expose contents; the resolver should interpret selectors.

ProblemThe agent owns too much domain behavior

The agent should manage goals, proposals, selection, and impasses. It should not be the place where “picking up an item means removing it from the world and adding it to me” is encoded as special behavior.

SmellFallbacks hide architecture bugs

A fallback like “if an object does not implement this, directly mutate its state dict” is convenient, but it makes invalid protocol usage look successful. That is dangerous in a generic NPC/agent architecture.

SmellThe rule library is already ahead of the engine

The crafting rule bundle emits effects such as ("Me", "consume", item) and ("Me", "create", output). Those are good abstractions, but they need real operation handlers, not more special cases added to ActionRule.apply.

3. North-star model

The clean conceptual split is:

Entities

Stable IDs, names, tags, components, inventory/container relationships, and state slots.

Selectors

Declarative references like name, tag, variable, current goal target, or scoped query.

Predicates

Truth checks such as Has, StateEquals, At, Contains, and GoalIs.

Action schemas

Parameterized actions with bindings, preconditions, and typed effects.

Operations

The only layer allowed to mutate world state.

A rule should not ask “does this Python object have fill()?” It should say: for some entity with tag Fluid_Transformer, set its contents to Water. Whether that entity is a Kettle object, a dict-backed ECS entity, or a remote game object should not matter.

4. Proposed concrete architecture

4.1 Entity and component model

Replace inheritance-heavy object semantics with a simple entity/component core. You can still keep convenience classes like Kettle or Mug, but they should mostly be constructors for entities with tags/components. 

from dataclasses import dataclass, field
from typing import Any

EntityId = str

@dataclass
class Entity:
    id: EntityId
    name: str
    tags: set[str] = field(default_factory=set)
    state: dict[str, Any] = field(default_factory=dict)
    components: set[str] = field(default_factory=set)

@dataclass
class WorldState:
    entities: dict[EntityId, Entity] = field(default_factory=dict)
    contains: dict[EntityId, set[EntityId]] = field(default_factory=dict)
    held_by: dict[EntityId, EntityId] = field(default_factory=dict)

Under this model:

  • Mug is an entity with tags/components like Drink_Vessel, Fillable, Sippable.
  • Kettle is an entity with Fluid_Transformer, Fillable, Heats_Water.
  • Cupboard is an entity with a Container component and a contains[cupboard] set.
  • Agent can also be an entity, which eliminates special handling for Me wherever possible.

4.2 Selector/resolver layer

Move iter_items, tag parsing, name parsing, scope search, and deduplication into a resolver. This is the right home for that logic. Not Rule, not Container, and not random helpers. 

class Selector:
    pass

@dataclass(frozen=True)
class ByName(Selector):
    name: str

@dataclass(frozen=True)
class ByTag(Selector):
    tag: str

@dataclass(frozen=True)
class Var(Selector):
    name: str

@dataclass
class Binding:
    values: dict[str, EntityId]

class Resolver:
    def resolve(self, world: WorldState, selector: Selector, binding: Binding) -> list[EntityId]:
        ...

Keep the string syntax like "tag:Drink_Vessel" only as a compatibility/frontend format. Parse it once into ByTag("Drink_Vessel"). Do not let raw string DSL leak into containers or object classes.

4.3 Predicate layer

Conditions should be typed predicates, not overloaded triples. 

@dataclass(frozen=True)
class Has:
    holder: Selector      # Me, World, container, location, etc.
    item: Selector

@dataclass(frozen=True)
class StateEquals:
    subject: Selector
    key: str
    value: object

@dataclass(frozen=True)
class Not:
    pred: object

@dataclass(frozen=True)
class GoalIs:
    pred: object

This gives you clean negation: 

Not(StateEquals(ByName("Me"), "at", "House"))

That is much cleaner than embedding negation into the predicate string as "NOT_at".

4.4 Action schemas with parameters

Actions should bind parameters first. Then preconditions and effects use those same bound variables. 

@dataclass
class ActionSchema:
    name: str
    params: dict[str, Selector]
    preconditions: list[object]
    effects: list[object]

Example: filling a kettle with water.

ActionSchema(
    name="FillKettleWater",
    params={
        "kettle": ByTag("Fluid_Transformer"),
        "source": ByTag("Dispenses_Water"),
    },
    preconditions=[
        Has(ByName("World"), Var("kettle")),
        Has(ByName("World"), Var("source")),
    ],
    effects=[
        SetState(Var("kettle"), "has", "Water"),
    ],
)

The important change is not cosmetic. The same bound kettle is used throughout the action. You no longer depend on multiple independent tag scans that coincidentally return the same object.

4.5 Typed operation/effect layer

Effects should become explicit operation objects. This is the biggest structural cleanup. 

@dataclass(frozen=True)
class SetState:
    subject: Selector
    key: str
    value: object

@dataclass(frozen=True)
class Transfer:
    item: Selector
    source: Selector
    target: Selector

@dataclass(frozen=True)
class RevealFromContainer:
    item: Selector
    container: Selector | None
    target: Selector

@dataclass(frozen=True)
class AddToSet:
    subject: Selector
    key: str
    value: object

@dataclass(frozen=True)
class Consume:
    item: Selector
    amount: int = 1

@dataclass(frozen=True)
class CreateEntity:
    template: str
    target: Selector

Then ActionRule.apply becomes boring: 

def apply_action(action: ActionSchema, world: WorldState, binding: Binding) -> list[Event]:
    events = []
    for effect in action.effects:
        events.append(operation_dispatch.apply(effect, world, binding))
    return events

Boring is good here. Once effects are typed, you do not need _coerce_effect, _default_apply_effect, or subject-specific branching inside the action rule.

5. Old tuple semantics → new operation semantics

Current form Hidden meaning Proposed typed form
("Me", "has", item) Pick up item from world into agent inventory. Transfer(item, source=World, target=Me)
("World", "retrieve", item) Find item inside a container and expose it to world. RevealFromContainer(item, container=Any(Container), target=World)
(drink_vessel, "has", "Hot_Water") Set the vessel contents to hot water. SetState(vessel, "has", "Hot_Water") or SetState(vessel, "contents", "Hot_Water")
("Me", "drank", "Coffee") Add Coffee to agent's drank set. AddToSet(Me, "drank", "Coffee") or fact Drank(Me, Coffee)
("Me", "NOT_at", "House") Agent is not at House. Not(StateEquals(Me, "at", "House"))
("Me", "consume", item) Remove or decrement an item/resource. Consume(item)
("Me", "create", output) Create a crafted item and put it in inventory. CreateEntity(template=output, target=Me)

6. Proposal rules should stay simple

Your proposal rules are conceptually fine: they map “current goal + world situation” to candidate actions. What I would change is their representation. Make proposals point to action schemas, not string action names, and optionally include a preference/priority. 

@dataclass
class ProposalRule:
    name: str
    when: list[object]
    propose: str
    priority: int = 0

Example:

ProposalRule(
    name="NeedHotWater_UseKettle",
    when=[
        GoalIs(StateEquals(ByTag("Drink_Vessel"), "has", "Hot_Water")),
        Has(ByName("World"), ByTag("Fluid_Transformer")),
    ],
    propose="BoilWater",
    priority=20,
)

The agent does not need to know how BoilWater works. It only needs to know that this proposal is relevant to the current goal. The action schema handles grounding and execution.

7. Three implementation options

Option AMinimal tuple cleanup

Keep tuple rules, but add a compiler that converts them into typed predicates and operations.

  • Fastest migration.
  • Good for current coffee/crafting experiments.
  • Still somewhat limited by legacy tuple shape.

Use if: you want working cleanup in a small number of files.

Option BAction schema core

Add typed selectors, predicates, binding, and operations. Keep object constructors as ergonomic wrappers.

  • Best balance of cleanliness and speed.
  • Rules and objects stop leaking into each other.
  • Extensible to village/search/crafting tasks.

Use if: you want the architecture to scale without a huge rewrite.

Option CFull fact/ECS core

Treat the whole world as a fact database plus entity components. Objects become views/templates.

  • Most generic and SOAR-like.
  • Best for a “digital brain” architecture.
  • Most disruptive rewrite.

Use if: the goal is a reusable general NPC architecture, not just coffee demos.

My recommendation is Option B now, with the data model shaped so it can become Option C later. This gives you immediate cleanup while avoiding the trap of designing an abstract universe before the engine has enough real examples.

8. Migration plan

  1. Add typed references and parser. Convert "tag:X", "Me", "World", and normal names into typed selectors at rule-load time.
  2. Add a resolver class. Move parse_tag, iter_items, find_item, has_item, and iter_subjects into this class. The resolver should be the only thing that knows how to scan scopes.
  3. Add typed predicates. Keep supporting old tuples temporarily, but compile them into Has, StateEquals, Not, and GoalIs.
  4. Add typed operations. Replace ActionRule.apply branches with an operation dispatcher.
  5. Delete fallback mutation. Unknown effect? Invalid. Unknown operation? Invalid. Object lacks required component? Action is not groundable/applicable.
  6. Move containers out of object/rule coupling. Containers should store entity IDs. Retrieval should be an operation on world/container relations, not a method that parses rule references.
  7. Ground actions before proposing/executing. Generate candidate action instances with concrete bindings. Let proposal selection choose among grounded actions or action schemas plus bindings.
  8. Add validation tests. Every rule bundle should compile. Every action should have known predicates/effects. Every effect should either apply or return a typed failure reason.

9. Tests and invariants to add

Compile-time validation

  • Reject unknown predicates like accidental "NOT_has" if it was meant as negation.
  • Reject unknown effects like ("Me", "craftify", x).
  • Reject effects with unbound variables.
  • Reject action schemas whose params are never used.

Runtime invariants

  • An entity should not be simultaneously in World and Me unless your model explicitly allows shared references.
  • Container contents should not also appear as visible world items unless revealed.
  • An effect should never silently skip unless the operation explicitly returns NoOp by design.
  • Tag-bound action variables should remain stable across preconditions and effects.

Regression cases

  • One mug, one kettle, hidden coffee: should succeed.
  • Multiple mugs/kettles: action bindings should be internally consistent.
  • No vessel in world/containers: should fail with a clear missing-resource reason.
  • Crafting bundle: consume and create should work without adding ad-hoc branches.

Debuggability

  • Log chosen proposal, chosen action schema, concrete binding, and operation events.
  • Log why each rejected action binding failed.
  • Keep final traces human-readable: “Transfer Mug from World to Me,” not “APPLY tag:Drink_Vessel.has=True.”

10. About removing iter_items from Rule

Yes: remove it from Rule. But do not move it to World as a random method either. Put it in a Resolver or QueryEngine.

Why? Because iter_items is not really a rule concept or a world concept. It is a reference-resolution concept. It needs to understand:

  • scope order: agent inventory vs visible world vs container contents vs locations;
  • selector type: name, tag, variable, component, current goal target;
  • deduplication: same entity visible through multiple paths;
  • binding consistency: same variable should keep pointing to the same entity;
  • visibility/knowledge restrictions: what the agent knows vs what the world actually contains.

A dedicated resolver lets you evolve all of that without contaminating the object classes or rule classes.

11. Sketch of the cleaned loop

def step(agent: AgentState, world: WorldState, rules: RuleBase):
    goal = agent.current_goal()

    # 1. Remove satisfied goals.
    while goal and evaluator.holds(goal, world, Binding({"Me": agent.entity_id})):
        agent.pop_goal()
        goal = agent.current_goal()

    # 2. Generate proposal candidates.
    proposals = []
    for proposal_rule in rules.proposals:
        for binding in grounder.bind_conditions(proposal_rule.when, world, agent, goal):
            proposals.append(ProposalInstance(proposal_rule, binding))

    # 3. Expand proposed action schemas into grounded actions.
    candidates = []
    for proposal in proposals:
        schema = rules.actions[proposal.propose]
        candidates.extend(grounder.ground_action(schema, world, agent, proposal.binding))

    # 4. Pick a candidate.
    choice = policy.select(candidates)

    # 5. If applicable, execute typed operations.
    if choice and evaluator.all_hold(choice.preconditions, world, choice.binding):
        return executor.apply(choice.effects, world, choice.binding)

    # 6. Otherwise, create subgoal from the most useful failed precondition.
    missing = diagnose_missing_precondition(choice, world)
    agent.push_goal(missing)

Notice what disappeared: no fallback mutation, no action-specific branches for Me.has, no container object parsing tag:, no coercing two-tuples into three-tuples during execution.

12. Final recommendation

The architecture should not ask every object to be clever. It should make the world model and operation system clever, and keep objects simple.

The most valuable refactor is: Resolver + typed predicates + typed operations + action schemas with bindings. Once that exists, your rules and objects can scale together instead of constantly needing compatibility glue.

I would not spend more time polishing the current tuple/fallback execution path. Treat it as a prototype that revealed the necessary abstractions. Preserve the nice parts — tags, rule bundles, goal proposals, object constructors — but move semantics into a single explicit protocol.

Source basis: reviewed the uploaded MakingCoffeeWithSoar.py and soar_rule_library.py. This proposal focuses on architecture and migration strategy rather than a drop-in patch.
